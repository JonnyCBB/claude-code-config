# Phase 5: Human Documentation Generation

This file contains the complete specification for Phase 5 of system-architecture-doc.
Read this file when Phase 5 begins.

Generate `README.md` with these sections:

## Mandatory Sections (always include)

**1. Overview**
- C4Context diagram (rendered SVG from Step 4d)
- Key facts table: system name, owner, tier, regions, component count

**2. Architecture**
- C4Container diagram(s) (rendered SVG from Step 4d) — if >10 elements or >12 relationships, define multiple views in workspace.dsl, each producing its own SVG.
- Container explanation table with columns: Container | Technology | Purpose | **Why It Exists**
- Communication patterns table: From → To | Protocol | Pattern (sync/async) | Purpose

**3. Key Request Flows** (2-3 flows)
Each flow follows the pattern: prose intro → `sequenceDiagram` → prose walkthrough

**4. Further Reading**
- Link to `workspace.dsl` (the enriched Structurizr DSL model)
- Links to TechDocs for each component (from Phase 2 inventory)
- Links to source documentation discovered in Phase 3

## Conditional Sections (include only if scope produced meaningful content)

**Component Deep-Dives** (max 1-2) — only for containers with complex internals discovered in Phase 3. Include a C4Component diagram (rendered SVG) if generated.

**Data** — only if data pipelines are in scope and Phase 2 discovered data stores/pipelines. Cover: storage systems, data flow, pipeline orchestration.

**Deployment and Infrastructure** — only if Phase 3 discovered meaningful deployment details (multi-region, scaling policies, etc.).

**Glossary** — only if domain-specific terms were encountered that wouldn't be obvious to a new team member.

## Sections NOT Auto-Generated

These require operational knowledge the agent can't reliably produce. List as human follow-up tasks in Phase 6:
- **Cross-Cutting Concerns**: auth patterns, observability, error handling strategies
- **Architecture Decisions**: ADR-style entries with context, decision, and consequences

## Design Constraints

- **5-8 diagrams total**, 20 elements max per diagram
- Every diagram: prose intro → diagram → prose walkthrough
- Active voice, specific, no filler
- Include a "Why It Exists" column in explanation tables
- Mark auto-generated content: `<!-- Auto-generated by system-architecture-doc — review before publishing -->`
- **PlantUML rendering**: All C4 structural diagrams (Context, Container, Component) are rendered as SVG images via the Step 4d pipeline. Mermaid is used only for `sequenceDiagram` blocks.
- **Split over clutter**: If a container diagram has >10 elements or >12 relationships, create multiple views in the `workspace.dsl` — each view produces a separate `.puml` → `.svg`.
- **Data stores**: Include in diagrams when they aid understanding (shared databases accessed by multiple services, caches that explain latency characteristics, queues that are central to the data flow). Omit when they only add clutter (per-service private stores, event buses used only for async logging, caches with a single consumer). The test: would removing this element cause a reader to misunderstand the architecture?
- **Footnotes for secondary edges**: Use `> **Note**: ...` prose footnotes below diagrams for secondary relationships not included in the SVG.
